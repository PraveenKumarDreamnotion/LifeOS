# 24 — Future Roadmap

> Everything the brief listed as "unsupported in the MVP", plus everything the MVP's own decisions deferred. Ordered by *what earns the right to exist next*, not by what is most exciting to build.
>
> **The rule that governs every item below:** if a feature would break the sentence *"LifeOS has no server, and nothing leaves your device by default,"* it must be opt-in, consented, and disclosed — or it does not ship.

## v0.2 — Trust and completeness (the debts the MVP knowingly took)

These are not features. They are repayments.

| Item | Why now | Cost |
| --- | --- | --- |
| **Auto-update** (`electron-updater` + GitHub `latest.yml`) | `latest.yml` is already generated by the MVP build. Users currently have no path to a bug fix. | 1 day |
| **Opt-in "Launch LifeOS when Windows starts"** | The single biggest limitation (`23` §1). `app.setLoginItemSettings()` writes one `HKCU` key, needs no elevation, uninstalls cleanly. **Off by default, explicit click required.** | 0.5 day |
| **Monthly recurrence** | `FREQ=MONTHLY;BYMONTHDAY=1`. The RRULE string is already stored, so **no data migration** — this is why we stored the string rather than three columns. Adopt `rrule-temporal` here. | 1 day |
| **Snooze for recurring reminders** | With an explicit choice: "just this once" or "move the whole schedule". | 0.5 day |
| **Export / Import JSON** | Also the prerequisite for any future sync. | 1 day |
| **Code signing** | Azure Trusted Signing ≈ $120/yr. Removes the SmartScreen warning for every user. | Money, not time |
| **NVDA accessibility pass** | `23` §9 is the section most likely to hide a real, fixable problem. | 1 day |

`MVP DECISION (forward-binding)` — v0.2 ships **no new capabilities**. It ships the things the MVP promised implicitly and did not deliver. A product that adds features before it can update itself is a product that cannot be fixed.

## v0.3 — Memory

The brief's §18, built at last. The schema already exists (migration 002); the feature does not.

```text
User:  "My grandfather has diabetes."
Yogi:  "Would you like me to remember that your grandfather has diabetes?"
                                                    [ Yes, remember ] [ No ]
       ...later...
User:  "What health conditions does my grandfather have?"
Yogi:  "You told me on 12 July that your grandfather has diabetes."
       > "My grandfather has diabetes."      ← the verbatim source utterance
```

### Non-negotiable constraints, decided now while they are free

1. **Memory obeys the same confirmation gate as reminders.** Yogi never stores a fact it inferred without asking. `source = 'user_confirmed'` is the only reachable value.
2. **A "What do you remember about me?" screen ships *before* any "remember this" feature.** Users must be able to read their profile before the app starts building one.
3. Every recalled memory displays the **verbatim source utterance** and its date. A mis-transcribed medical fact, recalled confidently six months later, is a real harm — and speech recognition of casual conversation is the least reliable input channel this app has.
4. `is_sensitive` is **derived, not chosen**: `category ∈ {health, family} ⇒ true`. Sensitive memories are never sent to any AI provider, even with AI Assist on. They are redacted from logs.
5. Per-row delete, always visible.

### Retrieval, in order of preference

| Approach | Verdict |
| --- | --- |
| **SQLite FTS5** | `RECOMMENDATION` — built into SQLite, no new dependency, no embeddings, no model. Handles "grandfather" + "health" trivially. Start here. |
| **Local embeddings** (`transformers.js` all-MiniLM) + `sqlite-vec` | `FUTURE OPTION` — only if FTS5 demonstrably fails on real queries. |
| Ollama embeddings | `FUTURE OPTION` — requires the user to install Ollama. |
| **LanceDB / ChromaDB** | ❌ A separate database process, for a few hundred facts. No. |

`MVP DECISION (forward-binding)` — **Do not reach for a vector database.** A personal second brain holds hundreds of facts, not millions. FTS5 answers "what health conditions does my grandfather have" perfectly and adds zero bytes to the installer. Reach for embeddings when a real query fails, and bring the failing query as evidence.

## v0.4 — A local LLM, and the end of the disclosure

The most interesting item on this roadmap, because it *deletes* complexity rather than adding it.

**Ollama behind the existing `AiAssistProvider` interface.**

```text
AiAssistProvider
├── OpenAiAssistProvider      (v0.1 — cloud, consented, disclosed)
└── OllamaAssistProvider      (v0.4 — local, free, no disclosure needed)
```

A local `llama3.2:3b` gives AI Assist's flexibility with **zero network, zero cost, and zero data leaving the device**. It collapses the entire consent apparatus in `09` §3 — the modal, the timestamp, the "last used" line, the CSP exception for `api.openai.com` — into nothing.

It is the *correct* long-term answer for a privacy-first product, and it was not in the MVP only because it requires the user to install Ollama separately (~2 GB), which no one will do to set a reminder.

`FUTURE OPTION` — With a local LLM in place, the following become defensible for the first time:

- Free-form conversation with Yogi
- Memory extraction from natural speech
- Multi-turn clarification that actually understands context
- Summarising a document the user opens

Each of these is a bad idea with a cloud model and a good idea with a local one. **That is the whole argument.**

## v0.5 — Research capability

The brief's §19. Modular provider interfaces, defined in the MVP (`core/research/`), implemented never.

```ts
export interface ResearchProvider {
  readonly id: string;
  readonly requiresNetwork: boolean;
  answer(query: string): Promise<ResearchResult>;
}
```

| Domain | Provider | Notes |
| --- | --- | --- |
| Weather | **Open-Meteo** | No API key, free, generous limits. The obvious first one. |
| Restaurants / places | Google Places, Foursquare | Keys, quotas, and a location disclosure. |
| Routes | OpenRouteService, Mapbox | Same. |
| Web research | A search API + LLM synthesis | Only worthwhile after v0.4. |
| PDF | Local text extraction (`pdf.js`) + optional local LLM summary | **Fully offline is achievable here.** |
| Medical | Curated sources + a strong disclaimer | See below. |
| Legal | Local document extraction + jurisdiction disclaimer | See below. |

### The rules for medical and legal

`MVP DECISION (forward-binding)` — LifeOS will never present itself as a source of medical, legal or financial advice.

1. It may summarise **a document the user gives it**. It may not answer *"do I have dengue?"*
2. Every such response opens with a disclaimer and closes with "speak to a qualified professional."
3. It cites sources. An uncited medical claim is not shown.
4. It never diagnoses, never recommends a dosage, never interprets a test result as good or bad.
5. Legal answers state their jurisdiction, or refuse.

`RISK (high, product)` — This is where a well-meaning personal assistant does real harm. The safest version of these features is the one that summarises what a document *says* and refuses to say what it *means*.

### The rule for every network provider

Each is an **individually consented, individually disclosed** toggle in Settings, off by default, added to the `connect-src` CSP and to the default-deny network allowlist only when enabled. Enabling "Weather" does not enable "Web research." Consent is not transitive.

## v1.0 — Only when it has earned it

`v1.0` is not a feature milestone. It is a claim: *the confirmation gate, the scheduler and the privacy guarantees have survived real users for a quarter.*

| Candidate | Precondition |
| --- | --- |
| **macOS / Linux** | `core/` is already pure TypeScript with no Electron imports. The port replaces `electron/` only. Notifications, tray, and TTS each need a platform implementation behind the existing interfaces. |
| **Mobile** | A rewrite, not a port. Flutter or React Native. Only worth it with sync. |
| **Opt-in E2EE sync** | ⚠️ **This is the feature that ends "LifeOS has no server."** It makes us a data controller under GDPR and India's DPDP Act (`22` §B5). It is a legal and organisational decision, not an engineering one. If it ships: end-to-end encrypted, keys on the device, server sees ciphertext, and the privacy statement is rewritten honestly. |
| Gmail / Calendar / WhatsApp | OAuth, app-review processes, and other people's data. Weeks each. |
| Wake word ("Hello Yogi") | An always-on KWS model. Requires a genuinely serious privacy design: a visible recording indicator, a local-only ring buffer, and a physical mute. It is not a small feature; it is a different product's trust model. |

## Permanently out of scope

Not "later." **Never.** These are not on a roadmap; they are the boundary that makes the roadmap trustworthy.

- Requiring administrator privileges
- Modifying Windows system files, drivers, services, or the registry beyond the uninstall entry
- Creating Windows Task Scheduler jobs
- Running shell or PowerShell commands constructed from user speech, user text, or model output
- Executing any code, script or command returned by an LLM
- Unrestricted desktop automation or input synthesis
- Deleting or modifying files outside the LifeOS data folder
- Silent self-update, silent software installation, or silent startup registration
- Autonomous agents that act without a confirmation gate
- Automatic purchases, messages, or calls
- Telemetry, analytics, or crash reporting that leaves the device without an explicit, per-event choice
- Uploading user data anywhere by default

`MVP DECISION` — The confirmation gate is the product's identity, not a safety feature bolted on. A version of LifeOS that acts autonomously is not a later version of LifeOS. It is a different product, and it should be given a different name.
